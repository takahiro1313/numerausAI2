## 1. システム構成

```
┌──────────────────────────────────────────────┐
│ 参加者ブラウザ（配布URL）                       │
│  Streamlit UI                                 │
│   ├─ 🔒 Step1 データプレビュー（全件）          │
│   ├─ 🟢 Step2 学習セル（手がかり＋model＋fit）→[学習]│
│   └─ 🟢 Step3 推論セル（predict）→ [推論]       │
└───────────────────┬──────────────────────────┘
                    │ 「学習/推論」押下（編集セルの文字列を送信）
                    ▼
   ┌───────────────────────────────────────┐
   │ Streamlit サーバ（Community Cloud）    │
   │  - ダミーCSV読込 → 翌月成約づくり・時間分割│
   │  - 限定名前空間で exec（学習セル/推論セル）│
   │  - 学習・推論（数秒）／スコア・重要度算出  │
   │  - 全担当予測を data/予測結果.csv に保存 ─┼─▶ ③まーぴーが読む
   └───────────────────────────────────────┘
  ※ 計算はサーバ側のみ。1人1URL=5＋予備1（デプロイ運用メモ）。
```

### 1.1 構成要素

| 要素 | 役割 | 技術 |
|---|---|---|
| Streamlit サーバ（Cloud） | 画面提供／データ管理／編集コードの安全実行 | Python, Streamlit |
| 数値AIモデル | 今月の活動→来月成約 の学習・推論 | XGBoost／線形回帰 |
| 学習セル | `手がかり`＋`model`作成＋`fit` を書く | `st.text_area` |
| 推論セル | `pred = model.predict(来月予測用)` を書く | `st.text_area` |
| 出力 | 予測値・train/valスコア対比・重要度 | `st.metric` / `st.bar_chart` |

### 1.2 設計の要点

- **手がかりを“活動の名前”で書く**：`手がかり = ["問い合わせ数", "面談数", ...]`。「AIに何を見せるか」を自分で決める＝直感的。
- **IT①との対比**：IT＝「人が if を書く」→ AI＝「AIがデータからルールを学ぶ」。
- **過学習＆手がかり実験**：時間分割で train/val を対比。手がかりから架電を外す／数字を上げると当たり方が変わる。
- **Cloud実行**：学習を役員PCで回さない。編集セルはサーバ側 `exec`。

## 3. 画面仕様（UI）

### 3.1 入力→処理→出力

| 段 | 内容 | ゾーン |
|---|---|---|
| 入力 | 担当×月の時系列（全件プレビュー） | 🔒 見るだけ（Step1） |
| 処理 | 手がかり → model作成 → fit（学習）／ predict（推論） | 🟢（Step2-3） |
| 反復 | 手がかり・数字を変えて再学習（過学習・効く活動を観察） | 🟢 |
| 出力 | 担当別の来月予測成約数 ＋ train/valスコア対比 ＋ 重要度 | 🔒 結果表示 |

### 3.2 編集ゾーン（🔒／🟢）

| 記号 | 該当 |
|---|---|
| 🔒 | データ読込・翌月成約づくり・時間分割・スコア計算・重要度グラフ・予測CSV保存 |
| 🟢 | Step2 学習セル（手がかり＋model＋fit）、Step3 推論セル（predict）、担当(selectbox) |

### 3.3 編集セルの初期スニペット（演習用／正解用）

🟢は **Step2 学習セル（model作成＋fit を1つ）** と **Step3 推論セル** の2つ。

**演習用（`app_ensyu.py`）：**
```python
# Step2（学習）
手がかり = ["___"]   # ← どの活動を見せる？ 列名を書く（例: "案件提案数", "面談数"）
model = XGBRegressor(n_estimators=40, max_depth=2, learning_rate=0.1, random_state=42)
model.fit(訓練データ[手がかり], 訓練データ["翌月の成約数"])
```
```python
# Step3（推論）
pred = model.predict(___)   # ← ___ に「来月予測用」を入れる
```
**正解用（`app_seikai.py`）：** 手がかり＝`["問い合わせ数","架電数","面談数","案件提案数"]`、`model.fit(訓練データ[手がかり], 訓練データ["翌月の成約数"])`、`pred = model.predict(来月予測用)`

> 🔒で名前空間に渡す：`訓練データ`（学習DF）／`来月予測用`（選んだ担当の最新月・手がかり列）／`XGBRegressor`／`LinearRegression`。

### 3.4 手がかり・モデル・過学習（🟢）

- **手がかり＝活動名のリスト**：`["問い合わせ数","架電数","面談数","案件提案数"]` から選ぶ。
- **モデルはコードで書く**：`XGBRegressor(n_estimators=, max_depth=, ...)`（木）／`LinearRegression()`（直線）。
- 体感できること：**架電だけ**にすると当たらない（val が落ちる）／**提案・面談**を入れると当たる。木の数字を上げると **train↑/val↓＝過学習**。
- 重要度は木モデルのみ（`feature_importances_` を選んだ手がかりで表示）。

## 4. 主要処理フロー

```
[参加者] Step1 データ確認（🔒・全件）
   │ Step2 学習セルに「手がかり＋model＋fit」を書く → [学習]
   ▼
[Cloud] 学習セルを限定名前空間で exec
   │   ns: XGBRegressor, LinearRegression, 訓練データ（→ 手がかり, model, fit）
   │ 成功 → 手がかり列で train/valスコア・重要度を表示／全担当予測を 予測結果.csv に保存
   │ 失敗 → 画面にエラー表示（落とさない）
   ▼
[参加者] Step3 担当を選び「pred = model.predict(来月予測用)」を書く → [推論]
   ▼
[Cloud] 推論セルを exec（ns: model, 来月予測用）→ pred → 来月予測成約数を表示
```

## 5. コンポーネント詳細（`app_ensyu.py`/`app_seikai.py`・1ファイル完結）

| 名称 | 役割 |
|---|---|
| `ACT_COLS` | 手がかりの候補＝活動4種（問い合わせ/架電/面談/案件提案） |
| `TARGET` | `"翌月の成約数"` |
| `DATA_PATH`/`PRED_PATH`/`VAL_MONTHS` | `activities.csv`／`予測結果.csv`／検証月数(3) |
| `SAFE_BUILTINS` | exec に渡す最小ビルトイン（import/IO は渡さない） |
| `build_features(df)` | 各担当の今月の活動に、答え＝翌月の成約数（`成約数`.shift(-1)）を付ける |
| `split_data(feat)` | 時間分割（過去で学習→未来で検証）。train_df/val_df/cutoff |
| `latest_row(df, rep)` | 選んだ担当の最新月（1行＝来月予測用）と直近3ヶ月（表示用） |
| `save_all_predictions(model, df, feature_cols)` | 全担当の来月予測を `予測結果.csv` に保存（🔒・連結） |
| `_exec_user(code, ns)` | 編集セルを限定名前空間で `exec` |
| `_do_learn` | [学習]：学習セルを exec → `手がかり`/`model` を取得、スコア算出・CSV保存 |
| `_do_predict` | [推論]：推論セルを exec → `pred` を取得 |

---

## 6. 安全な実行（exec）設計

| 項目 | 方針 |
|---|---|
| 名前空間の限定 | `exec(code, {"__builtins__": SAFE_BUILTINS}, ns)`。学習時 `XGBRegressor`/`LinearRegression`/`訓練データ`（＋生成された `手がかり`/`model`）、推論時 `model`/`来月予測用` のみ |
| I/O 遮断 | `import`/ファイル/ネットワークI/Oは渡さない |
| 例外捕捉 | `try/except` で全例外を捕捉し**画面表示**（落とさない）。手がかりの列名ミスは `KeyError` で捕捉 |
| 前提 | 信頼できる少人数＋インスタンス分離（1人1URL） |

|---|---|
| 外部通信 | なし |

## 8. エラー処理

| 事象 | 挙動 |
|---|---|
| 手がかりの列名ミス／空欄 `___` | `KeyError` 等を捕捉し画面表示。`app_seikai.py` で答え合わせ |
| model 未作成・未学習で推論 | 「先に学習してください」 |
| 数字を極端に上げる | 学習は実行し、過学習として train/val 差に表れる（警告） |
| CSV未配置 | `python make_dummy.py` で生成 |
